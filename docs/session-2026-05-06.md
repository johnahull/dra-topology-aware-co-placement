# Session Notes — 2026-05-06

## KubeVirt Multi-NUMA VM Demo with DRA Topology Coordinator

### Goal

Create a demo deploying a KubeVirt VM spanning 2 NUMA nodes on the XE8640 (4x H100 SXM5), with all devices (GPU, CPU, memory) allocated via DRA topology coordinator partition device classes. The guest should see correct NUMA topology with GPUs on the correct NUMA nodes.

### Demo: `demos/kubevirt-dra-topology-gpu/`

**Files:**
- `demo-vm-2numa.yaml` — 2 ResourceClaims (quarter-numa0, quarter-numa1) + VirtualMachine
- `configmaps.yaml` — topology coordinator rules (with `deviceFilter: vfio`)
- `xe8640-kubevirt-2numa.tape` — VHS recording script
- `xe8640-kubevirt-2numa.gif` — terminal recording
- `README.md` — demo documentation

**VM Configuration:**
- 2 sockets × 4 cores = 8 vCPUs
- 8 GiB guest memory (hugepages 2Mi)
- `guestMappingPassthrough` for NUMA topology
- 2 GPUs via `gpus` field referencing `partition-vfio-gpu-nvidia-com` request
- Fedora 41 container disk with cloud-init (password: fedora)
- 4 GiB reserved overhead for VFIO DMA mapping

**Result:**
- Guest sees 2 NUMA nodes: node0 (CPUs 0-3, ~4GB), node1 (CPUs 4-7, ~4GB)
- H100 GPU on each NUMA node with correct affinity
- `pxb-pcie` expander bridges place PCI devices on correct guest NUMA nodes

### Bugs Found and Fixed

#### TC-3: Double GPU count in partition device classes

The topology coordinator counted both `gpu-0` (compute) and `gpu-vfio-0` (VFIO) as separate devices from `gpu.nvidia.com`, allocating 2 GPUs per quarter instead of 1.

**Root cause:** `driverCounts` in `buildProportionalPartitions` counted all devices from a driver without filtering by device type.

**Fix (3 parts):**
1. Added `deviceFilter` field to topology rule ConfigMaps — filters devices by attribute value (e.g., `deviceFilter: vfio` for `gpu.nvidia.com/type`)
2. Fixed qualified attribute name matching — unqualified DRA attributes (e.g., `type`) now match rules with qualified names (e.g., `gpu.nvidia.com/type`)
3. Removed unconditional `divided=1` fallback for exclusive devices that can't be subdivided

**Files changed:**
- `k8s-dra-topology-coordinator/internal/controller/partition_builder.go`
- `k8s-dra-topology-coordinator/internal/controller/topology_model.go`
- `k8s-dra-topology-coordinator/internal/controller/topology_rules.go`

#### TC-4: Unqualified attribute name mismatch

`extractTopologyDevice()` compared raw attribute names (`type`) against rule attribute names (`gpu.nvidia.com/type`) literally, so topology rules never matched unqualified attributes.

**Fix:** Construct qualified form (`driverName/name`) for unqualified attributes and try both when matching.

#### KV-11: Misleading "no pciBusID" error from multi-device requests

KubeVirt's `resolveDevice()` rejects requests with >1 device, but the error was swallowed by `createHostDeviceForGPU()`, producing a misleading "has no mdevUUID or pciBusID" message. Root cause was TC-3 (2 GPUs per partition). Fixed by fixing TC-3.

### Infrastructure Notes

- **virtctl v1.8.2** installed on XE8640 — `virtctl ssh` broken due to server version `$Format:%h$` not being valid semver. The KubeVirt fork was built without proper ldflags version injection.
- **NVIDIA DRA driver** kubelet plugin shows 1/2 ready — the `compute-domains` container stays not-ready on all-VFIO systems (no nvidia compute driver). This is expected behavior.
- **`dedicatedCpuPlacement`** removed from VM spec — requires `kubevirt.io/cpumanager=true` node label which virt-handler overwrites based on kubelet's `cpuManagerPolicy`. With Option A (`cpuManagerPolicy: none`), the label stays `false`. DRA CPU driver handles pinning via NRI regardless.
- **`kubevirt.io/cpumanager` label** — virt-handler continuously reconciles this label from kubelet config. Manual overrides get reverted.
